# Ключевые архитектурные решения

## 1. Единый источник истины

### Проблема

При раздельном хранении геометрии для IFC и для 3D-визуализации возникает риск рассогласования.

### Решение

Геометрия строится **один раз** в IFC-модели, затем конвертируется в mesh.

```
IFC геометрия (источник)
        │
        ▼
  Конвертация
        │
        ▼
  Three.js mesh (представление)
```

### Преимущества

- Гарантия соответствия 3D-вида и IFC-данных
- Отсутствие дублирования кода построения геометрии
- Упрощение тестирования

## 2. RepresentationMaps + IfcMappedItem

### Проблема

Многократное дублирование геометрии для одинаковых компонентов увеличивает размер IFC.

### Решение

Использование механизма переиспользования геометрии IFC:

```
IfcMechanicalFastenerType
└── RepresentationMaps → IfcRepresentationMap
    └── MappedRepresentation → IfcShapeRepresentation

IfcMechanicalFastener (экземпляр)
└── ObjectPlacement → IfcLocalPlacement
└── Representation → IfcProductDefinitionShape
    └── Representations → IfcMappedItem
        └── MappingSource → IfcRepresentationMap
```

### Преимущества

- Геометрия хранится один раз на уровне типа
- Экземпляры содержат только трансформацию
- Уменьшение размера IFC в 5–10 раз

## 3. Динамическое кэширование типов

### Проблема

Создание нового типа для каждого болта увеличивает размер IFC.

### Решение

Кэширование типов в памяти с ключом:

```python
key = (bolt_type, execution, diameter, length, material)
```

### Алгоритм

```python
def get_or_create_type(params):
    key = make_key(params)
    if key in cache:
        return cache[key]
    else:
        type_entity = create_type(params)
        cache[key] = type_entity
        return type_entity
```

### Преимущества

- Повторное использование типов для одинаковых болтов
- Оптимизация размера IFC
- Ускорение генерации

## 4. Работа в браузере (Pyodide)

### Проблема

Традиционно IfcOpenShell требует установки Python и компиляции.

### Решение

Использование Pyodide (Python WebAssembly) для выполнения Python в браузере.

### Архитектура

```
Браузер
├── JavaScript (UI, Three.js)
└── Pyodide (WebAssembly)
    ├── ifcopenshell
    ├── numpy
    └── Python модули приложения
```

### Преимущества

- Без установки ПО
- Кроссплатформенность
- Прямой доступ к IfcOpenShell

### Ограничения

- Время загрузки ~80 МБ
- Ограничения памяти браузера
- Отсутствие файлового доступа

## 5. DI-контейнер

### Проблема

Управление зависимостями между Python-модулями.

### Решение

Простой DI-контейнер с ленивой инициализацией:

```python
class DIContainer:
    def __init__(self, ifc_doc):
        self.ifc_doc = ifc_doc
        self._cache = {}

    def get_geometry_builder(self):
        if 'geometry_builder' not in self._cache:
            self._cache['geometry_builder'] = GeometryBuilder(self.ifc_doc)
        return self._cache['geometry_builder']
```

### Преимущества

- Централизованное управление зависимостями
- Кэширование экземпляров
- Упрощение тестирования (mock)

## 6. Двухэтапная инициализация

### Проблема

Pyodide требует времени на загрузку и инициализацию.

### Решение

Разделение на этап загрузки и этап генерации:

| Этап | Действия | Время |
|------|----------|-------|
| 1. Загрузка | Pyodide, зависимости, модули | ~5 сек |
| 2. Генерация | Валидация, построение, конвертация | ~1 сек |

### Преимущества

- Чёткое разделение ответственности
- Возможность прогресс-индикации
- Оптимизация повторных генераций
